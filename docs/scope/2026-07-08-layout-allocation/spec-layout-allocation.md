# Layout Allocation

## 目标

当前竖屏布局采用固定底部分析区高度，窄竖屏下容易浪费棋盘空间。本次调整要把布局改成先确定顶部、底部分析区的最小需求，再把剩余空间分配给棋盘和操作区；在保证胜率/曲线区完整显示、数据区可用的前提下，优先放大棋盘并保持布局稳定。

## 决策

- 顶部 bar 和标题区域保持现状，不重新设计。
- 底部分析区按竖屏画布宽度判断单列/双列：`<700px` 为单列，`>=700px` 为双列。
- 双列底部分析区的最小高度必须保证左侧胜率/曲线区完整显示。
- 单列底部分析区的最小高度为“胜率/曲线区完整高度 + 数据区至少一行可见高度”，数据区内部滚动，不撑开整体布局。
- 底部分析区高度是最小高度，不是固定限制；棋盘和操作区排布完成后，剩余高度继续分配给底部分析区。
- 主体先从总高度扣除顶部和底部最小高度，剩余空间分配给棋盘和操作区。
- 如果剩余高度能容纳“满宽棋盘 + 横向操作区”，棋盘横向铺满并向上靠近标题/对局信息区，操作区紧贴棋盘下方，底部分析区位于操作区下方并获得剩余高度。
- 如果剩余高度不足以容纳满宽棋盘加横向操作区，则棋盘在中部显示，操作区作为竖向栏贴棋盘右侧；棋盘和右侧操作栏作为整体居中。

## 架构

布局由三个稳定区域组成：顶部工具/标题区、主体棋盘操作区、底部分析区。顶部保持现有 DOM 和视觉；底部分析区先由宽度断点决定单列或双列，并提供最小高度；主体棋盘操作区使用剩余高度，通过现有布局测量流程计算可用矩形并选择“棋盘满宽 + 下方操作栏”或“棋盘居中 + 右侧操作栏”。

## 流程

布局计算顺序为：先确定竖屏画布尺寸，再保留顶部自然高度，然后根据底部单列/双列规则得到底部最小高度，最后把剩余空间交给棋盘和操作区。棋盘区域在剩余空间内选择最大可用正方形；当满宽棋盘能和横向操作栏同时放入时采用横向操作栏，并让棋盘在主体区域内向上靠、紧挨标题/对局信息区，横向操作栏紧贴棋盘下方；当满宽棋盘不能和横向操作栏同时放入时采用右侧竖向操作栏。主体区域用掉棋盘和操作区所需高度后，剩余高度归底部分析区，底部分析区在其内部滚动。

## 验收标准

- 横屏设备仍使用竖屏画布，不引入横屏布局。
- `<700px` 竖屏画布使用底部单列分析区，且至少完整显示胜率/曲线区和一行数据区。
- `>=700px` 竖屏画布使用底部双列分析区，左侧胜率/曲线区完整显示，右侧详情区与左侧等高并内部滚动。
- 棋盘区域使用扣除顶部和底部最小高度后的剩余空间，不被底部数据列表内容撑小。
- 底部分析区高度不固定为最小高度；在棋盘和操作区完成排布后，剩余垂直空间由底部分析区承接。
- 剩余空间足够高时，棋盘横向铺满并向上靠近标题/对局信息区，操作区紧贴棋盘下方，操作区下方紧贴底部分析区。
- 剩余空间不足以容纳满宽棋盘加横向操作区时，棋盘和右侧竖向操作栏作为组合居中。
- 数据变化、列表变长、窗口 resize、移动端系统栏变化时，整体布局不发生内容撑开的跳动。
